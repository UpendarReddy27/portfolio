01 About

I design and implement pipelines that move data from REST APIs and operational sources into the cloud and the warehouse. That usually means Python and SQL, Apache Airflow on Docker (with PostgreSQL and Redis when the architecture needs them), Amazon S3 for raw and curated files, AWS Lambda and EventBridge for event-driven steps, and Apache Spark when the workload needs distributed processing. I focus on reliable orchestration, clear transformations, and outputs that analysts and downstream jobs can rely on.

In Snowflake I work in layered patterns (raw → staging → marts), dimensional modeling, RBAC, and reporting SQL — including external stages, Snowpipe, tasks, retention policies, zero-copy cloning, and warehouse sizing. Alongside batch and warehouse work, I build MCP servers (including a PostgreSQL MCP server for Cursor) so AI assistants interact through named tools instead of unconstrained database access — pairing disciplined data engineering with developer-facing integrations.

M.S. Computer Science, Bradley University (2023–2025); B.S. Computer Science, Bharath University (2019–2023).

Layers · dimensions · RBAC

PostgreSQL MCP server

MCP · PostgreSQL · Python

MCP tooling: a small Model Context Protocol server that connects an AI assistant (for example in Cursor) to PostgreSQL. Exposes named tools to list tables, run read queries, and perform basic CRUD-style operations. Configure with DATABASE_URL or separate DB_* variables in .env; runs over stdio for MCP hosts (FastMCP, psycopg2, python-dotenv). Includes a security note: use least-privilege DB users and trusted data, since fragments are interpolated into SQL.
